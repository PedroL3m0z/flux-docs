Esta página leva você do zero até uma conta conectada enviando uma mensagem. Cada passo aponta para seu guia completo.

## 1. Suba o gateway

A Flux vem como uma **imagem única** que junta API, PostgreSQL e Redis em um só
container. Suba das duas formas — ambas rodam a mesma imagem:

<Tabs>
  <TabItem label="docker run">

Baixe e rode a imagem publicada — sem clonar nada:

```bash
# Docker Hub
docker run -d -p 3000:3000 -v flux_data:/data pedrooaj/flux-api

# ou GitHub Container Registry
docker run -d -p 3000:3000 -v flux_data:/data ghcr.io/pedrol3m0z/flux-api
```

  </TabItem>
  <TabItem label="docker-compose.yml">

Escreva seu próprio `docker-compose.yml` apontando para a imagem publicada, com
um volume nomeado para `/data`. Defina as credenciais inline (ou deixe de fora
para gerar automaticamente):

```yaml
# docker-compose.yml
services:
  flux:
    image: pedrooaj/flux-api
    ports:
      - "3000:3000"
    volumes:
      - flux_data:/data
    environment:
      SEED_USERNAME: admin
      SEED_PASSWORD: ch4nge-me
      API_KEY: my-api-key
    restart: unless-stopped

volumes:
  flux_data:
```

Depois suba:

```bash
docker compose up -d
```

  </TabItem>
</Tabs>

A imagem é construída para `linux/amd64` e tem tag por release (`X.Y.Z`, `X.Y`,
`latest`). O volume `/data` persiste o banco, os dados do Redis e os segredos
gerados automaticamente — **mantenha-o** entre recriações do container. Um só
container, sem isolamento de serviços: um restart recicla todos os processos —
ideal para single-host / self-hosting.

| Superfície | URL |
| --- | --- |
| API | `http://localhost:3000` |
| Dashboard | `http://localhost:3000/dashboard` |
| Docs interativas da API (Scalar) | `http://localhost:3000/docs` |

:::tip[Configuração zero]
Nenhuma variável de ambiente é necessária. No primeiro boot a Flux gera segredos fortes e um usuário **admin** inicial, imprimindo a **API key** e a **senha** do admin no log **uma única vez** (`docker logs <container>`) — copie agora. Eles também ficam persistidos no volume `/data` (`secrets.json`).
:::

### Defina suas próprias credenciais (opcional)

Em vez de copiar os valores gerados do log, você pode escolher o **usuário**, a
**senha** do admin e a **API key** de antemão com variáveis de ambiente. O que
ficar vazio continua sendo gerado automaticamente.

| Variável | Define | Padrão se vazio |
| --- | --- | --- |
| `SEED_USERNAME` | usuário de login do admin | `admin` |
| `SEED_PASSWORD` | senha de login do admin | gerada, impressa no log uma vez |
| `SEED_EMAIL` | email do admin | gerado |
| `API_KEY` | o valor do `x-api-key` | gerada, impressa no log uma vez |

<Tabs>
  <TabItem label="docker run">

Passe com `-e`:

```bash
docker run -d -p 3000:3000 -v flux_data:/data \
  -e SEED_USERNAME=admin \
  -e SEED_PASSWORD='ch4nge-me' \
  -e API_KEY='my-api-key' \
  pedrooaj/flux-api
```

  </TabItem>
  <TabItem label="Docker Compose">

Coloque num `.env` ao lado do `docker-compose.yml` (ele lê o arquivo
automaticamente):

```bash
# .env
SEED_USERNAME=admin
SEED_PASSWORD=ch4nge-me
API_KEY=my-api-key
```

  </TabItem>
</Tabs>

Eles semeiam **apenas** o primeiro boot — criam o usuário admin e os segredos e
depois ficam persistidos em `/data` (`secrets.json`). Mudá-los depois **não**
reescreve uma conta existente; para resetar, limpe o volume ou troque a senha no
dashboard. Trate `SEED_PASSWORD` / `API_KEY` como segredos — não os commite.

## 2. Faça login

Toda requisição protegida precisa de duas coisas: um **JWT** (quem você é) e a **API key** (`x-api-key`). Obtenha o JWT fazendo login:

```bash
curl -X POST http://localhost:3000/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin","password":"<do log de boot>"}'
# → { "accessToken": "<JWT>" }
```

Veja [Autenticação](/flux-docs/pt-br/authentication/) para o quadro completo.

## 3. Defina suas credenciais do Telegram

A Flux fala com o Telegram usando um `api_id` / `api_hash` de [my.telegram.org](https://my.telegram.org):

```bash
curl -X PUT http://localhost:3000/telegram/settings \
  -H 'Authorization: Bearer <JWT>' \
  -H 'x-api-key: <API_KEY>' \
  -H 'Content-Type: application/json' \
  -d '{"apiId":"123456","apiHash":"abcdef..."}'
```

Mais em [Credenciais do Telegram](/flux-docs/pt-br/telegram-credentials/).

## 4. Crie uma instância

```bash
curl -X POST http://localhost:3000/telegram/instances \
  -H 'Authorization: Bearer <JWT>' -H 'x-api-key: <API_KEY>' \
  -H 'Content-Type: application/json' \
  -d '{"label":"Conta principal"}'
# → { "id": "<instanceId>", "status": "new", ... }
```

Veja [Instâncias](/flux-docs/pt-br/instances/).

## 5. Autorize a instância (login no Telegram)

Abra um stream de login por QR e escaneie o código com o celular, ou faça login por número de telefone + código. Fluxos completos em [Sessões](/flux-docs/pt-br/sessions/).

## 6. Envie uma mensagem

```bash
curl -X POST \
  http://localhost:3000/telegram/instances/<instanceId>/chats/<chatId>/messages \
  -H 'Authorization: Bearer <JWT>' -H 'x-api-key: <API_KEY>' \
  -H 'Content-Type: application/json' \
  -d '{"text":"Olá da Flux"}'
```

A partir daqui, assine [Webhooks](/flux-docs/pt-br/webhooks/) ou consuma [Eventos](/flux-docs/pt-br/events/) para reagir a mensagens recebidas.

## Prefere o dashboard?

Tudo acima também é point-and-click em `http://localhost:3000/dashboard`: faça login, defina credenciais em **Settings**, crie uma instância, conecte-a via QR ou telefone, e abra seus chats.

![A página Overview do dashboard, anotada](../../../assets/screenshots/overview-annotated.png)

O **Overview** mostra uptime, contagem de instâncias e webhooks, e a saúde dos serviços num relance. A página **Help** dentro do app percorre o mesmo fluxo passo a passo.
